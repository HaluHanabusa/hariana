# hariana(針穴)

**hariana = 針穴(はりあな)。** 大量の情報を層のように重ねたトンネルに
針を通す — その針が通る穴が、誰もまだいない価値ある場所(Whitespace)です。

新しい製品や研究テーマの「まだ誰もやっていない場所(Whitespace)」を、
AIエージェントに探させるためのフォルダ一式です。
Claude Code・Codex・Cursor などのローカルAIエージェントに読み込ませて使います
(実行手順は業界標準の AGENTS.md 形式。ファイルの読み書きとWeb検索が
できるエージェントなら動きます)。
分からない言葉が出てきたら GLOSSARY.md を見てください。

## どう動くのか(5つのフェーズ)
1. **壁打ち** — AIが質問して、テーマと守るべき条件(制約)を一緒に固める
2. **調査** — AIがWebの一次情報だけを集める。推論や記憶で数値を書くのは禁止。
   「探したが無かった」ことも検索式つきで記録する
3. **行列** — 集めた事実を表(行=アイデア、列=条件)に整理する。
   考え方の型(Skill)が行を生み、条件の合否を判定する
4. **針通し** — 有望に見える候補を、特許の裏取りと「red-team」(わざと攻撃する係)
   の審査にかける。攻撃は物理・お金・過去の失敗・前提・
   「なぜ今まで誰もやっていないのか」の5方向。大半はここで死ぬ
5. **企画書** — 生き残った候補だけを、あなたの立場に合う企画書
   (週末MVP要件 / 研究計画書)に自動で仕上げる

## 他のツールと違う3つの点
- **型の掛け合わせ = 表の充足**。TRIZやジョブ理論などの「考え方の型」を
  順番に唱えるのではなく、1つの表の別々の列を埋める係として協働させる
- **red-teamは取り外せない**。アイデアの9割を殺す前提の設計。
  生き残りには必ず「残存リスク一覧」が付く(リスクゼロ判定は差し戻し)
- **空白には2種類ある**。「作れる製品の座標」が全滅しても、
  「世の中に測定やデータが無い場所」= 研究テーマとして拾えるので、
  探索が空振りで終わらない

## はじめ方
SETUP.md を読んでください(所要5分)。要点は3行:
```bash
bash scripts/init.sh individual   # 立場に合うプロファイルで初期化(Codexなら第2引数に codex)
claude                            # 起動(codex や Cursor でも可 — SETUP.md 手順4)
# 「〇〇というテーマで探索を始めたい。Phase 0からお願いします」と話す
```
夜間も回し続けたい場合は AUTOPILOT.md へ。

## フォルダ構成
| 場所 | 中身 | AIから書ける? |
|---|---|---|
| DESIGN_CHARTER.md | 設計の憲法(思想・やらないこと・決定事項) | 不可 |
| AGENTS.md / AUTOPILOT.md | 実行手順(正本 / 無人運転用)。CLAUDE.md 等の入口は init が生成 | 不可 |
| Skill_Library/ | 考え方の型の保管庫 | 不可 |
| Active_Skills/ | 今回使う型のコピー(自由に注釈してよい) | 可 |
| core/red-team/ | 攻撃係+過去の失敗DB(graveyard) | 不可 |
| profiles/ | 立場別の設定(制約・型・企画書形式・しきい値) | 不可 |
| templates/ | 企画書の雛形(空白の類型で使い分け) | 不可 |
| output/ | あなたの探索データ(Obsidianで開ける。git管理外) | 可 |
| tests/ | 答えの分かっている検証問題と結果 | 不可 |

「不可」の場所は、Claude Code では `.claude/settings.json` で書き込み禁止を
強制しています(AIが自分のルールを書き換えられない構造)。強制機構の無い
エージェント(Codex / Cursor 等)では、AGENTS.md 冒頭の「書き込み境界」の
指示と git 履歴が防壁になります。

## 実例: 「見つからない」を正直に報告した話
賃貸ベランダ菜園の「旅行中10日間の水やり」をテーマに通し運転をしたところ、
AIは出典つきの事実54件を集め、アイデア3件を作り、そして**3件すべてを
自分の攻撃係(red-team)で却下しました**。理由は具体的です — 1件は中核部品が
物理的に成立せず放棄特許まで存在、1件は業界全社が避ける保証リスクを
「差別化」と勘違い、1件は需要の仮説自体が反証された。
成熟市場で「空白はありません。理由はこれです」と言えること、
これがこのツールの中核の保証です(詳細: tests/e2e-run-01-results.md)。
