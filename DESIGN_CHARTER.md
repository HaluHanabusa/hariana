# DESIGN CHARTER — hariana(針穴)

このドキュメントは、実装より先に「何であるか / 何でないか」を固定するための憲章。
以降の全ての設計判断はここに従う。変更する場合は理由を追記して履歴を残す。

## 一言での定義

> フレームワークを「適用」するのではなく、制約を「充足」させることで
> 構造的な空白座標(Whitespace)を発見するイノベーション探索OSS。
> 自分のPCと自分のAI契約(BYOK)だけで動き、運営サーバーを持たない。

## コア思想(差別化の源泉)

### 1. 制約充足による発見 (Constraint Satisfaction, not Prompt Chaining)
思考フレームワークは順番に通す「バケツリレー」ではなく、
**共有されたConstraint Matrixの異なる列を埋める役割**として実装する。

| フレームワーク | 埋める列 |
|---|---|
| ジョブ理論 | 顧客制約(未充足ジョブの有無) |
| TRIZ | 技術矛盾(解決パターンの適用可否) |
| ブルーオーシャン戦略 | 市場飽和(競合密度・減らせる価値要素) |
| フルーガル革新 | コスト・資源制約 |
| 第一原理思考 | 物理・原理限界(行の分解と再構成) |

Whitespace = 全列が PASS または UNKNOWN の行。
フレームワークを増やす = 列を増やすこと。この定義により拡張が単調になる。

### 2a. Whitespaceの2類型(ドメイン運用からの抽出、2026-07-16確定)
実際の探索運用(Claim約100件規模)で、Whitespaceには2つの型があることが
確認された(4事例で再発、2回ルールを充足):
- **設計座標型**: Matrixの全列が非FAILの行 = 作れる製品・システムの座標
- **研究ギャップ型**: 構造的な情報空白そのもの(計測ギャップ/モデル化ギャップ/
  業界的非開示構造)が研究貢献になる座標。全列PASS行が全滅した盤面でも成立する
両類型とも針通し(特許裏取り→red-team)は必須。企画書テンプレートは
類型で分岐する(設計座標型=事業企画、研究ギャップ型=研究計画/論文構想)。

### 2. Whitespaceの3値判定(幻覚防止の中核規律)
「検索してヒットしない」は発見ではなく、判定の開始点である:
- (a) 未発見 — 本物の空白。研究・事業候補
- (b) 不可能 — 物理か経済が禁じている。KILL
- (c) 死屍 — 試されて死んだ。死因が時期的(前提技術の変化)な場合のみ蘇生候補

red-team-critic相当のフェーズを通過しない候補は、いかなる出力にも含めない。

### 3. 日本語の実務語彙が第一級市民
TRIZの40発明原理、助成金プロポーザル(JSPS/NEDO/AMED)、社内起案書、
特定商取引法などを日本語の実務語彙で扱う。英訳READMEは提供するが、
Skill本文は日本語を正とする。欧米OSS(Orchestra, ARIS等)との市場的差別化。

## Non-goals(やらないこと)

1. **コード生成をしない** — 企画書(Markdown)を出力して終わる。
   実装はClaude Code / Cursor等の領域であり、企画書がそのバトンになる
2. **実験実行をしない** — GPU実験や自動評価はOrchestra/ARISの領域
3. **SaaS化しない** — BYOKローカル固定。運営コストゼロ、機密が外に出ない
4. **チャットUIを作らない** — ObsidianとローカルエージェントのCLIで完結する
5. **Skillに任意コード実行を持ち込まない** — Skillは純Markdown(指示+知識)を原則とし、
   スクリプトが必要な場合は本体リポジトリの監査済みscripts/のみ参照する
   (Skillエコシステムのハイジャック攻撃事例を踏まえたセキュリティ方針)

## アーキテクチャ決定(AD)

- **AD-01 プロファイルはYAML、Gitブランチは使わない**
  個人/中小企業/大企業/研究者の差は `profiles/*.yml`(初期制約・使用Skill・
  出力テンプレ)で表現する。ブランチはstable/devのリリース管理のみ。
  理由: 4ブランチ維持はメンテコスト4倍であり、修正のチェリーピック地獄になる
- **AD-02 状態はObsidian Vault(Markdown + frontmatter + wikilink)**
  Claim / Matrix / Whitespace候補 / 企画書のすべてをファイルとして永続化。
  グラフビューが「どの事実からどのアイデアが生まれたか」の系譜になる
- **AD-03 着脱は Skill_Library → Active_Skills へのコピー + .gitignore**
  (原構想を採用。git pull時のコンフリクトゼロを保証)
- **AD-04 ドメインインスタンスがケーススタディ**
  本フレームワークの機能追加・汎用化は、必ず「実際に動いたドメイン運用」
  からの抽出で行う(逆方向は禁止)。ドメインインスタンスは公開・非公開を
  問わず実証事例として扱う
- **AD-05 HITLは「コミットメント」の裁定者であり、位置はプロファイルで可変**
  人間が必ず握るのは「どの候補に時間と資金を投じるか」の決定。
  ドラフト(企画書化)は可逆で安価なため、red-team通過候補の企画書化は
  AIが自動で行ってよい。`hitl_gate: pre-proposal | post-proposal` を
  プロファイルで選択する(個人=post推奨、大企業=pre推奨)。
  ただし無人運転での注意喚起の洪水を防ぐため、1サイクルあたりの
  企画書生成上限(ランク付き、既定3本)を必須とする。
  red-team通過は人間ゲートではなく機械ゲートとして常に必須

- **AD-06 敵対的レビュー(red-team)は着脱不可のコア機構**
  Skill_Library(着脱式)ではなく core/ に置き、Active_Skillsの選択に
  関係なく常時適用する。企画書化の前提条件であり、ユーザーが外す手段を
  提供しない。red-teamは代替案を提案しない(却下専用)ことで攻撃の鈍化を防ぐ

- **AD-07 実行手順の正本はAGENTS.md(エージェント非依存)**
  コミットされる実行手順は業界横断標準の AGENTS.md の1本のみ。エージェント
  固有の入口(Claude Code用 CLAUDE.md、旧Cursor用 .cursor/rules/)は、
  scripts/init.sh が bootstrap/agents/ のテンプレートから生成する
  gitignore済みの薄いポインタとし、選ばなかったエージェントの入口は作らない
  (手順の複製・分岐の禁止。AD-03と同じ「生成物はgit管理外」の原則)。
  本システムのSkillは製品機能ではなく「読ませる純Markdown」の規約(Non-goal 5)
  なので、ファイル読み書きとWeb検索ができるエージェントなら追加実装なしで動く。
  書き込み境界の機械的強制(.claude/settings.json)が使えないエージェントでは、
  AGENTS.md冒頭の「書き込み境界」の指示とgit履歴を代替の防壁とする。
  理由: エージェント別に手順書を複製すると修正が即座に分岐し、AD-01で
  ブランチを退けたのと同じメンテ地獄になる(2026-07-18、マルチエージェント対応時に追加)

## しきい値の扱い
本文書および各ドキュメントに現れる数値しきい値(企画書上限、Claim蓄積数、
2回ルール等)はすべて既定値であり、`profiles/*.yml` の thresholds で
調整可能とする。固定値をコードやSkillに埋め込むことを禁じる。
