# ジョブ理論 (Jobs to be Done)

## このSkillが呼ばれる場面
- 行(設計選択肢)が「技術的に作れる」段階に達し、「誰かが雇うか」の判定が必要なとき
- 機能競争(スペック追加合戦)に行が収束し、視点の転換が必要なとき

## 手順

### Step 1: 利用状況の特定
行が想定する具体的な利用場面を書く。**セグメントはデモグラフィック
(年齢・職業)ではなく状況で切る**。「通勤中の朝」「工事現場の雨天時」のように。

### Step 2: ジョブの3面分解
その状況で片付けたいジョブを分解する:
- **機能的ジョブ** — 物理的に達成したいこと
- **感情的ジョブ** — どう感じたいか(安心・達成感・退屈しのぎ)
- **社会的ジョブ** — 他者からどう見られたいか

### Step 3: ジョブステートメント化
「**[状況]のとき、[動機]したい。そうすれば[期待成果]**」の形式で記述する。
この形式にならない「ジョブ」は、ジョブではなく機能の言い換えである。

### Step 4: 代替手段の列挙(最重要)
現行の代替をすべて列挙する。**競合製品とは限らない** — 別カテゴリの製品、
人手、回避行動、そして「何もしない(非消費)」も代替である。
代替の列挙が競合製品だけで終わっている分析は差し戻す。

### Step 5: 充足度判定
- **PASS**: 特定状況で現行代替が明確にジョブを満たせていない(under-served)、
  または過剰供給で「減らした」代替が雇われうる(over-served)。根拠Claim必須
- **FAIL**: 現行代替で十分に足りている(行が雇われる理由がない)
- **UNKNOWN**: 充足度の証拠がない → 調査タスク化(想像で埋めない)

## 出力契約
- ジョブステートメント、3面分解、代替リスト(非消費含む)、判定根拠Claim IDを必須とする
- 「もっと高性能なら売れる」型の結論は本Skillの失敗とみなす(ジョブ不在の兆候)
