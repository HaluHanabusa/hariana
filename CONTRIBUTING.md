# CONTRIBUTING — 貢献の手引き

hariana への関心をありがとうございます。個人メンテのプロジェクトなので
返信に数日かかることがあります。日本語が第一言語ですが、英語のIssue/PRも歓迎です。

## いちばん欲しい貢献: 「使ってみた報告」

本プロジェクトは「実際に動いたドメイン運用からの抽出」でしか機能を増やしません
(DESIGN_CHARTER.md AD-04)。なので「〇〇というテーマで回した/ここで詰まった/
この判定は変だった/Whitespaceが1本生き残った」という報告が、そのまま開発の
燃料になります。**うまくいかなかった報告のほうが貴重です。**
Issueテンプレート「事例・フィードバック報告」からどうぞ。

注意: output/ の抜粋を貼るときは、公開したくない情報(未公開のアイデア・
事業計画など)が含まれていないか確認してください。

## Issue

- バグ報告・質問・提案はなんでも歓迎です。用語は GLOSSARY.md 参照
- ただし「このアイデアは有望ですか?」という相談には答えません
  (それはあなたの環境で動く red-team の仕事です)

## Pull Request

歓迎する領域と、それぞれの条件:

| 領域 | 条件 |
|---|---|
| Skill_Library/ への新Skill | Skill_Library/README.md の契約に従う(frontmatterで role/column を宣言、判定基準を本文に明記、例は2ドメイン以上に跨る、先に tests/validation-problems.md へ検証問題を追加) |
| core/red-team/graveyard/ への死屍事例 | 出典URL必須。死因分類(技術的/経済的/時期的)と蘇生条件(revivable)を明記 |
| profiles/ の新プロファイル | 実在の立場に対応すること(例: sme / enterprise)。しきい値には理由を添える |
| templates/ の企画書テンプレート | Whitespace 2類型のどちらに対応するかを明記 |
| ドキュメント改善・誤字修正・翻訳 | そのままどうぞ |

受け入れられないPR(先に謝っておきます):

- red-team の着脱化・緩和(DESIGN_CHARTER.md AD-06。設計の中核です)
- Skillへの実行コードの持ち込み(Non-goal 5。Skillは純Markdownが原則)
- 出典のない数値・事例の追加(Phase 1の規律と同じ基準をリポジトリ本体にも適用)
- 実運用の裏付けがない新機能(AD-04。まず「事例・フィードバック報告」として
  出してください。2回以上再発したパターンだけを汎用化します)

設計判断の基準はすべて DESIGN_CHARTER.md にあります。大きめの変更は、
先にIssueで相談してもらえると互いに時間を無駄にしません。
